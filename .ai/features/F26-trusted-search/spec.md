# F26 — Trusted Search

## Goal

Make AI search genuinely reliable by (1) fixing constraint enforcement and ranking, (2) returning a structured API response that exposes what was understood, found, relaxed, and why — all from deterministic code.

## Background

Current shortcomings traceable to code:

1. **Constraint leak**: `_relax_filters` relaxes `bedrooms_min` and `bathrooms_min`, which `constraint-policy.md` classifies as hard constraints.
2. **Ranking is set-merge**: `_resolve_result_ids` produces unordered merged sets, not scored ranking.
3. **Strict/relaxed results mixed**: API returns one flat `items` list; users cannot distinguish complete matches from broadened ones.
4. **Relaxation disclosure unstructured**: `relaxed_conditions` is `list[str]`; frontend cannot render structured chips.
5. **No per-result match explanation**: no information about why a specific result appeared.
6. **Soft-constraint relaxation not implemented**: `_relax_filters` currently has no logic for `subway_distance_max` or `built_year_min` — these are soft constraints per policy but are never actually relaxed.

References:
- `.ai/features/F20-relaxation-layer/constraint-policy.md`
- BUG-025: `parsed_filters` preserves original intent
- BUG-022: compound queries enter pipeline

## Scope

### Backend — `backend/app/services/ai_search/service.py`

**1. Fix `_relax_filters` (two changes)**

Remove the `bedrooms_min` block (lines 622–632) and the `bathrooms_min` block (lines 634–644) — these are hard constraints and must never be auto-relaxed.

Add new relaxation logic for soft constraints (currently missing):
- `subway_distance_max`: when set, remove it (soft constraint, remove entirely)
- `built_year_min`: when set, remove it (soft constraint, remove entirely)
- `location`: keep existing removal logic (conditional soft, no change)

New relaxation order: `subway_distance_max` → `built_year_min` → `location`

**2. Hybrid scoring in `_resolve_result_ids`**

Replace the current set-merge with score fusion:

```python
score_map = defaultdict(float)
for i, pid in enumerate(semantic_ids):
    score_map[pid] += 1.0 / (i + 1)   # semantic rank signal
for pid in filter_ids:
    score_map[pid] += 0.8              # filter match bonus
result_ids = sorted(score_map, key=lambda pid: score_map[pid], reverse=True)
```

**3. Hard-filter revalidation**

Add `_matches_hard_constraints(property: PropertyRead, filters: SearchFilters) -> bool` that checks all hard constraint fields (price, bedrooms, bathrooms, area). A result may only appear in `strict_items` if it passes this check.

**4. Produce structured output in `ai_search()`**

- `strict_items`: results where all hard AND soft constraints are satisfied (revalidated)
- `recommended_items`: results where all hard constraints are satisfied but ≥1 soft constraint was relaxed
- `relaxations: list[RelaxationRecord]`: what changed and to what value
- `parsed_constraints: list[ConstraintInfo]`: structured version of parsed_filters for frontend rendering
- `match_reasons: dict[str, list[MatchReason]]`: keyed by property id; generated by deterministic code, not LLM

### Backend — `backend/app/schemas/ai_search.py`

New types:

```python
class RelaxationRecord(BaseModel):
    field: str       # e.g. "subway_distance_max"
    from_value: Any  # original value, or None if was not set
    to_value: Any    # relaxed value, or None if removed

class MatchReason(BaseModel):
    field: str       # e.g. "bedrooms_min"
    label: str       # e.g. "≥ 4 bedrooms"
    matched: bool
    strength: Literal["hard", "soft"]

class ConstraintInfo(BaseModel):
    field: str
    value: Any
    strength: Literal["hard", "soft"]
    label: str       # e.g. "≤ 2000万"
```

Updated `AiSearchResult`:

```python
class AiSearchResult(SearchResult):
    # Existing fields (unchanged, backwards compat)
    parsed_filters: SearchFilters
    ai_summary: str
    query_parsed: bool
    # items: inherited from SearchResult
    # total: inherited from SearchResult
    # page / page_size: inherited

    # New fields
    parsed_constraints: list[ConstraintInfo]
    strict_items: list[PropertyRead]
    recommended_items: list[PropertyRead]
    relaxations: list[RelaxationRecord]
    match_reasons: dict[str, list[MatchReason]]  # key = property.id
```

**`items` contract**: `items = strict_items + recommended_items`, strict first, recommended after. `total = len(strict_items) + len(recommended_items)`. This is computed and set by `ai_search()` after producing the two lists.

**`match_reasons` placement**: a dict keyed by property id. This avoids introducing a new envelope type for `PropertyRead` and keeps backwards compat. The frontend looks up reasons by `property.id`. If this proves inconvenient in practice, a future feature can migrate to an item-level envelope.

### Frontend — `frontend/components/features/search/`

- Two-zone result layout: "完全符合条件 (N套)" above, "推荐匹配 (N套)" below with relaxation notice
- Relaxation notice above recommended zone, rendered from `relaxations` (e.g. "已放宽：地铁距离限制已移除")
- `match_reasons` chips on each card (hard reasons in one color, soft in another; unmatched reasons with △)
- `ai-parsed-filters-card.tsx`: render from `parsed_constraints` when present (shows strength: hard/soft)

### Evaluation — `backend/tests/`

New files: `eval_set_f26.json` + `test_eval_f26.py`

**Minimum 10 queries, required coverage:**

| # | Query type | What it validates |
|---|---|---|
| 1–2 | Hard constraint only (bedrooms + price) | `strict_items` all pass hard-filter revalidation |
| 3–4 | Soft constraint only (subway/built_year) | relaxation triggers, `recommended_items` populated |
| 5–6 | Mixed hard + soft | correct split between strict and recommended |
| 7 | Chinese vocabulary (四房 近地铁 新楼) | pipeline entry + parsing |
| 8 | Query that triggers no relaxation (enough results) | `relaxations` is empty, `recommended_items` empty |
| 9 | Query that yields 0 results even after relaxation | empty both lists, no crash |
| 10 | Non-search query | `query_parsed=False`, `strict_items` and `recommended_items` empty |

**Acceptance conditions:**

- **Hard-filter precision**: no result in `strict_items` violates any hard constraint (bedrooms_min, bathrooms_min, price_min, price_max, area_min, area_max). Assert 0 violations across all test queries.
- **strict/recommended split correctness**: if `relaxations` is non-empty, `strict_items` must be the pre-relaxation result set; `recommended_items` is the post-relaxation supplement. Assert this relationship holds for queries 3–6.
- **Relaxation correctness**: for each `RelaxationRecord`, `from_value` matches the original `parsed_filters` value; `to_value` matches what was actually applied. Assert for queries 3–6.
- **No regression**: existing `eval_set.json` / `test_eval.py` (F16) still passes 30/30.
